# 설명서

## 명령이 수락됐는지 확인한 방법

앱은 커피 시작 명령을 보낸 뒤 바로 끝났다고 판단하지 않습니다.  
명령 레지스터를 다시 읽습니다.

- `0x0000`이면 머신이 명령을 받은 상태입니다.
- `0xFFFF`이면 머신이 명령을 거부한 상태입니다.

거부되면 앱은 같은 주문을 최대 3번 다시 시도합니다.  
그 결과, 순간적으로 머신이 바빠도 주문이 바로 사라지지 않습니다.

## 제조 완료를 판단한 방법

앱은 컵이 빠졌다는 이유만으로 완료 처리하지 않습니다.  
반드시 아래 순서가 맞아야 정상 완료입니다.

1. 제조 완료 신호 `RCP_STATE=2` 확인
2. 그 다음 컵 빠짐 `CUP_STATUS=0` 확인

이 순서가 중요한 이유는 컵이 중간에 빠질 수 있기 때문입니다.  
완료 신호 전에 컵이 빠지면 정상 완료가 아니라 이상 상황으로 표시합니다.

## 여러 잔 주문 처리 방법

앱 안에 대기열을 둡니다.  
사용자가 3잔을 주문해도 머신에는 한 번에 1잔만 보냅니다.

한 잔이 `제조 완료 -> 컵 빠짐` 순서로 끝나면 다음 잔을 자동으로 시작합니다.  
그래서 머신이 동시에 여러 명령을 받아 꼬이는 일을 막습니다.

## 통신이 끊겼을 때 처리 방법

응답이 없으면 앱은 멈추지 않습니다.  
짧은 지연이면 화면에 `응답 지연`을 표시하고 연결은 유지합니다.  
이 상태에서는 다음 응답이 오면 바로 정상 표시로 돌아옵니다.

응답 없음이 여러 번 이어지면 `응답 없음`으로 바꾸고 연결을 다시 엽니다.  
그 결과, `fault 3`처럼 일부러 통신이 흔들리는 상황에서도 앱이 바로 죽거나 주문을 쉽게 잃지 않습니다.

Modbus 요청은 순간 실패에 대비해 짧게 재시도합니다.  
그래서 한 번 늦은 응답 때문에 제조 흐름이 바로 실패 처리되지 않습니다.
